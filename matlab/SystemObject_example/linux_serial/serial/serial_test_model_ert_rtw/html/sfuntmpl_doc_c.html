42/* Error handling



43 * --------------



44 *



45 * You should use the following technique to report errors encountered within



46 * an S-function:



47 *



48 *       ssSetErrorStatus(S,"error encountered due to ...");



49 *       return;



50 *



51 * Note that the 2nd argument to ssSetErrorStatus must be persistent memory.



52 * It cannot be a local variable in your procedure. For example the following



53 * will cause unpredictable errors:



54 *



55 *      mdlOutputs()



56 *      {



57 *         char msg[256];         {ILLEGAL: to fix use "static char msg[256];"}



58 *         sprintf(msg,"Error due to %s", string);



59 *         ssSetErrorStatus(S,msg);



60 *         return;



61 *      }



62 *



63 * The ssSetErrorStatus error handling approach is the suggested alternative



64 * to using the mexErrMsgTxt function.  MexErrMsgTxt uses "exception handling"



65 * to immediately terminate S-function execution and return control to



66 * Simulink. In order to support exception handling inside of S-functions,



67 * Simulink must setup exception handlers prior to each S-function invocation.



68 * This introduces overhead into simulation.



69 *



70 * If you do not call mexErrMsgTxt or any other routines that cause exceptions,



71 * then you should use SS_OPTION_EXCEPTION_FREE_CODE S-function option.  This



72 * is done by issuing the following command in the mdlInitializeSizes function:



73 *



74 *      ssSetOptions(S, SS_OPTION_EXCEPTION_FREE_CODE);



75 *



76 * Setting this option, will increase the performance of your S-function by



77 * allowing Simulink to bypass the exception handling setup that is usually



78 * performed prior to each S-function invocation.  Extreme care must be taken



79 * to verify that your code is exception free when using the



80 * SS_OPTION_EXCEPTION_FREE_CODE option.  If your S-function generates



81 * an exception when this option is set, unpredictable results will occur.



82 *



83 * Exception free code refers to code which never "long jumps". Your S-function



84 * is not exception free if it contains any routine which when called has



85 * the potential of long jumping. For example mexErrMsgTxt throws an exception



86 * (i.e. long jumps) when called, thus ending execution of your S-function.



87 * Use of mxCalloc may cause unpredictable problems in the event of a memory



88 * allocation error since mxCalloc will long jump. If memory allocation is



89 * needed, you should use the stdlib.h calloc routine directly and perform



90 * your own error handling.



91 *



92 * All mex* routines have the potential of long jumping (i.e. throwing an



93 * exception). In addition several mx* routines have the potential of



94 * long jumping. To avoid any difficulties, only the routines which get



95 * a pointer or determine the size of parameters should be used. For example



96 * the following will never throw an exception: mxGetPr, mxGetData,



97 * mxGetNumberOfDimensions, mxGetM, mxGetN, mxGetNumberOfElements.



98 *



99 * If all of your "run-time" methods within your S-function are exception



100 * free, then you can use the option:



101 *      ssSetOptions(S, SS_OPTION_RUNTIME_EXCEPTION_FREE_CODE);



102 * The other methods in your S-function need not be exception free. The



103 * run-time methods include any of the following:



104 *    mdlGetTimeOfNextVarHit, mdlOutputs, mdlUpdate, and mdlDerivatives



105 *



106 * Warnings & Printf's



107 * -------------------



108 *   You can use ssWarning(S,msg) to display a warning. 



109 *    - When the S-function is compiled via mex for use with Simulink,



110 *      ssWarning equates to mexWarnMsgTxt.



111 *    - When the S-function is used with Real-Time Workshop, 



112 *      ssWarning(S,msg) equates to 



113 *        printf("Warning: in block '%s', '%s'\n", ssGetPath(S),msg);



114 *      if the target has stdio facilities, otherwise it becomes a comment and 



115 *      is disabled.



116 *   



117 *   You can use ssPrintf(fmt, ...) to print a message. 



118 *    - When the S-function is compiled via mex for use with Simulink,



119 *      ssPrintf equates to mexPrintf.



120 *    - When the S-function is used with Real-Time Workshop, 



121 *      ssPrintf equates to printf, if the target has stdio facilities, 



122 *      otherwise it becomes a call to a empty function (rtPrintfNoOp).



123 *    - In the case of Real-Time Workshop which may or may not have stdio



124 *      facilities, to generate the most efficient code use:



125 *         #if defined(SS_STDIO_AVAILABLE)



126 *            ssPrintf("my message ...");



127 *         #endif



128 *    - You can also use this technique to do other standard I/O related items,



129 *      such as:



130 *         #if defined(SS_STDIO_AVAILABLE)



131 *             if ((fp=fopen(file,"w")) == NULL) {



132 *                ssSetErrorStatus(S,"open failed");



133 *                return;



134 *             }



135 *             ...



136 *         #endif



137 */

403/* Function: mdlInitializeSizes ===============================================



404 * Abstract:



405 *    The sizes information is used by Simulink to determine the S-function



406 *    block's characteristics (number of inputs, outputs, states, etc.).



407 *



408 *    Direct Feedthough:



409 *       The direct feedthrough flag can be either 1=yes or 0=no. It should be



410 *       set to 1 if the input, "u", is used in the mdlOutput or



411 *       mdlGetTimeOfNextVarHit() function. Setting this to 0 is akin to making



412 *       a promise that "u" will not be used in the mdlOutput or



413 *       mdlGetTimeOfNextVarHit() function. If you break the promise, then



414 *       unpredictable results will occur.



415 *



416 *       It is very common for S-function authors to write incorrect S-functions



417 *       when configuring the S-function direct feedthrough flag. We often find



418 *       that S-function authors are confused about what the correct setting for



419 *       the direct feedthrough flag should be. Part of the confusion is because



420 *       the term direct feedthrough is misleading. To reduce the confusion, you



421 *       can think of the direct feedthrough setting as a 'needs input'



422 *       setting. Specifically, if the S-function access an input signal in



423 *       either mdlOutputs() or mdlGetTimeOfNextVarHit(), then the direct



424 *       feedthrough flag must be set to 1 (true).



425 *    



426 *       For example, if a level 2 C-MEX S-function uses:



427 *          ssGetInputPortSignal(S,inputPortIndex)



428 *       in its mdlOutputs() or mdlGetTimeOfNextVarHit() methods, then the 



429 *       S-function is required to set the direct feedthrough flag to true in 



430 *       its mdlInitializeSizes() method:



431 *          ssSetInputPortDirectFeedThrough(S, inputPortIdx, 1);



432 *    



433 *       If your S-Function uses ssGetInputPortSignal() in mdlOutputs() or



434 *       mdlGetTimeOfNextVarHit() and fails to specify



435 *       the direct feedthrough, or specifes it wrongly using



436 *         ssSetInputPortDirectFeedThrough(S, inputPortIdx, 0);



437 *       then the S-Function is incorreclty written and Simulink will not



438 *       operate correctly and may crash.



439 *



440 *       When you compile your S-function with debugging, e.g.,



441 *         mex -g sfunction_name.c



442 *       your S-function will be instrumented such that an incorrect access to



443 *       an input signal will generate a diagnostic message.



444 *



445 *    The NumContStates, NumDiscStates, NumInputs, NumOutputs, NumRWork,



446 *    NumIWork, NumPWork NumModes, and NumNonsampledZCs widths can be set to:



447 *       DYNAMICALLY_SIZED    - In this case, they will be set to the actual



448 *                              input width, unless you have a mdlSetWorkWidths



449 *                              to set the widths.



450 *       0 or positive number - This explicitly sets item to the specified



451 *                              value.



452 */



453static void mdlInitializeSizes(SimStruct *S)



454{



455    int_T nInputPorts  = 1;  /* number of input ports  */



456    int_T nOutputPorts = 1;  /* number of output ports */



457    int_T needsInput   = 1;  /* direct feed through    */



458



459    int_T inputPortIdx  = 0;



460    int_T outputPortIdx = 0;



461



462



463    ssSetNumSFcnParams(S, 0);  /* Number of expected parameters */



464    if (ssGetNumSFcnParams(S) != ssGetSFcnParamsCount(S)) {



465        /*



466         * If the the number of expected input parameters is not equal



467         * to the number of parameters entered in the dialog box return.



468         * Simulink will generate an error indicating that there is a



469         * parameter mismatch.



470         */



471        return;



472    }



473



474    /* 



475     * Configure tunability of parameters.  By default, all parameters are



476     * tunable (changeable) during simulation.  If there are parameters that 



477     * cannot change during simulation, such as any parameters that would change 



478     * the number of ports on the block, the sample time of the block, or the 



479     * data type of a signal, mark these as non-tunable using a call like this:



480     * 



481     *    ssSetSFcnParamTunable(S, 0, 0);



482     *



483     * which sets parameter 0 to be non-tunable (0).



484     *



485     */



486



487



488    /* Register the number and type of states the S-Function uses */



489



490    ssSetNumContStates(    S, 0);   /* number of continuous states           */



491    ssSetNumDiscStates(    S, 0);   /* number of discrete states             */



492



493



494    /*



495     * Configure the input ports. First set the number of input ports. 



496     */



497    if (!ssSetNumInputPorts(S, nInputPorts)) return;    



498    /*



499     * Set input port dimensions for each input port index starting at 0.



500     * The following options summarize different ways for setting the input 



501     * port dimensions. 



502     *



503     * (1) If the input port dimensions are unknown, use



504     *     ssSetInputPortDimensionInfo(S, inputPortIdx, DYNAMIC_DIMENSION))



505     *



506     * (2) If the input signal is an unoriented vector, and the input port 



507     *     width is w, use



508     *     ssSetInputPortVectorDimension(S, inputPortIdx, w)



509     *     w (or width) can be DYNAMICALLY_SIZED or greater than 0.



510     *     This is equivalent to ssSetInputPortWidth(S, inputPortIdx, w).



511     *



512     * (3) If the input signal is a matrix of dimension mxn, use 



513     *     ssSetInputPortMatrixDimensions(S, inputPortIdx, m, n)



514     *     m and n can be DYNAMICALLY_SIZED or greater than zero.



515     *



516     * (4) Otherwise use:



517     *     ssSetInputPortDimensionInfo(S, inputPortIdx, dimsInfo)



518     *     This function can be used to fully or partially initialize the port 



519     *     dimensions. dimsInfo is a structure containing width, number of 



520     *     dimensions, and dimensions of the port.



521     */



522    if(!ssSetInputPortDimensionInfo(S, inputPortIdx, DYNAMIC_DIMENSION)) return;



523    /*



524     * Set direct feedthrough flag (1=yes, 0=no).



525     * A port has direct feedthrough if the input is used in either



526     * the mdlOutputs or mdlGetTimeOfNextVarHit functions.



527     * See sfuntmpl_directfeed.txt.



528     */



529    ssSetInputPortDirectFeedThrough(S, inputPortIdx, needsInput);



530



531    /*



532     * Configure the output ports. First set the number of output ports.



533     */



534    if (!ssSetNumOutputPorts(S, nOutputPorts)) return;



535



536    /*



537     * Set output port dimensions for each output port index starting at 0.



538     * See comments for setting input port dimensions.



539     */



540    if(!ssSetOutputPortDimensionInfo(S,outputPortIdx,DYNAMIC_DIMENSION)) return;



541



542    /*



543     * Set the number of sample times. This must be a positive, nonzero



544     * integer indicating the number of sample times or it can be



545     * PORT_BASED_SAMPLE_TIMES. For multi-rate S-functions, the



546     * suggested approach to setting sample times is via the port



547     * based sample times method. When you create a multirate



548     * S-function, care needs to be taking to verify that when



549     * slower tasks are preempted that your S-function correctly



550     * manages data as to avoid race conditions. When port based



551     * sample times are specified, the block cannot inherit a constant



552     * sample time at any port.



553     */



554    ssSetNumSampleTimes(   S, 1);   /* number of sample times                */



555



556    /*



557     * Set size of the work vectors.



558     */



559    ssSetNumRWork(         S, 0);   /* number of real work vector elements   */



560    ssSetNumIWork(         S, 0);   /* number of integer work vector elements*/



561    ssSetNumPWork(         S, 0);   /* number of pointer work vector elements*/



562    ssSetNumModes(         S, 0);   /* number of mode work vector elements   */



563    ssSetNumNonsampledZCs( S, 0);   /* number of nonsampled zero crossings   */



564



565    /* Specify the sim state compliance to be same as a built-in block */



566    /* see sfun_simstate.c for example of other possible settings */



567    ssSetSimStateCompliance(S, USE_DEFAULT_SIM_STATE);



568



569    /*



570     * All options have the form SS_OPTION_<name> and are documented in



571     * matlabroot/simulink/include/simstruc.h. The options should be



572     * bitwise or'd together as in



573     *   ssSetOptions(S, (SS_OPTION_name1 | SS_OPTION_name2))



574     */



575



576    ssSetOptions(          S, 0);   /* general options (SS_OPTION_xx)        */



577



578} /* end mdlInitializeSizes */

796/* Function: mdlInitializeSampleTimes =========================================



797 * Abstract:



798 *



799 *    This function is used to specify the sample time(s) for your S-function.



800 *    You must register the same number of sample times as specified in



801 *    ssSetNumSampleTimes. If you specify that you have no sample times, then



802 *    the S-function is assumed to have one inherited sample time.



803 *



804 *    The sample times are specified as pairs "[sample_time, offset_time]"



805 *    via the following macros:



806 *      ssSetSampleTime(S, sampleTimePairIndex, sample_time)



807 *      ssSetOffsetTime(S, offsetTimePairIndex, offset_time)



808 *    Where sampleTimePairIndex starts at 0.



809 *



810 *    The valid sample time pairs are (upper case values are macros defined



811 *    in simstruc.h):



812 *



813 *      [CONTINUOUS_SAMPLE_TIME,  0.0                       ]



814 *      [CONTINUOUS_SAMPLE_TIME,  FIXED_IN_MINOR_STEP_OFFSET]



815 *      [discrete_sample_period,  offset                    ]



816 *      [VARIABLE_SAMPLE_TIME  ,  0.0                       ]



817 *



818 *    Alternatively, you can specify that the sample time is inherited from the



819 *    driving block in which case the S-function can have only one sample time



820 *    pair:



821 *



822 *      [INHERITED_SAMPLE_TIME,  0.0                       ]



823 *    or



824 *      [INHERITED_SAMPLE_TIME,  FIXED_IN_MINOR_STEP_OFFSET]



825 *



826 *    The following guidelines may help aid in specifying sample times:



827 *



828 *      o A continuous function that changes during minor integration steps



829 *        should register the [CONTINUOUS_SAMPLE_TIME, 0.0] sample time.



830 *      o A continuous function that does not change during minor integration



831 *        steps should register the



832 *              [CONTINUOUS_SAMPLE_TIME, FIXED_IN_MINOR_STEP_OFFSET]



833 *        sample time.



834 *      o A discrete function that changes at a specified rate should register



835 *        the discrete sample time pair



836 *              [discrete_sample_period, offset]



837 *        where



838 *              discrete_sample_period > 0.0 and



839 *              0.0 <= offset < discrete_sample_period



840 *      o A discrete function that changes at a variable rate should



841 *        register the variable step discrete [VARIABLE_SAMPLE_TIME, 0.0]



842 *        sample time. The mdlGetTimeOfNextVarHit function is called to get



843 *        the time of the next sample hit for the variable step discrete task.



844 *        Note, the VARIABLE_SAMPLE_TIME can be used with variable step



845 *        solvers only.



846 *      o Discrete blocks which can operate in triggered subsystems.  For your 



847 *        block to operate correctly in a triggered subsystem or a periodic 



848 *        system it must register [INHERITED_SAMPLE_TIME, 0.0]. In a triggered



849 *        subsystem after sample times have been propagated throughout the



850 *        block diagram, the assigned sample time to the block will be 



851 *        [INHERITED_SAMPLE_TIME, INHERITED_SAMPLE_TIME]. Typically discrete



852 *        blocks which can be periodic or reside within triggered subsystems



853 *        need to register the inherited sample time and the option



854 *        SS_OPTION_DISALLOW_CONSTANT_SAMPLE_TIME. Then in mdlSetWorkWidths, they



855 *        need to verify that they were assigned a discrete or triggered



856 *        sample time. To do this:



857 *          mdlSetWorkWidths:



858 *            if (ssGetSampleTime(S, 0) == CONTINUOUS_SAMPLE_TIME) {



859 *              ssSetErrorStatus(S, "This block cannot be assigned a "



860 *                               "continuous sample time");



861 *            }



862 *



863 *    If your function has no intrinsic sample time, then you should indicate



864 *    that your sample time is inherited according to the following guidelines:



865 *



866 *      o A function that changes as its input changes, even during minor



867 *        integration steps should register the [INHERITED_SAMPLE_TIME, 0.0]



868 *        sample time.



869 *      o A function that changes as its input changes, but doesn't change



870 *        during minor integration steps (i.e., held during minor steps) should



871 *        register the [INHERITED_SAMPLE_TIME, FIXED_IN_MINOR_STEP_OFFSET]



872 *        sample time.



873 *



874 *    To check for a sample hit during execution (in mdlOutputs or mdlUpdate),



875 *    you should use the ssIsSampleHit or ssIsContinuousTask macros.



876 *    For example, if your first sample time is continuous, then you



877 *    used the following code-fragment to check for a sample hit. Note,



878 *    you would get incorrect results if you used ssIsSampleHit(S,0,tid).



879 *        if (ssIsContinuousTask(S,tid)) {



880 *        }



881 *    If say, you wanted to determine if the third (discrete) task has a hit,



882 *    then you would use the following code-fragment:



883 *        if (ssIsSampleHit(S,2,tid) {



884 *        }



885 *



886 */



887static void mdlInitializeSampleTimes(SimStruct *S)



888{



889    /* Register one pair for each sample time */



890    ssSetSampleTime(S, 0, CONTINUOUS_SAMPLE_TIME);



891    ssSetOffsetTime(S, 0, 0.0);



892



893} /* end mdlInitializeSampleTimes */
